## v0.4.1 (next hotfix)

### 1. Dashboard footer shows `—` for model + agent during running experiment

**Reported:** 2026-05-02 (Mike's Mac + Ubuntu, fresh 0.4.0 install,
running stroop-test).

**Symptom:**

```
⠴ Evaluating…
model
—
·
agent
—
·
tokens 109K
·
cost  $3.08
```

The token/cost counters update correctly. The `model` and `agent`
fields stay at the placeholder dash even while a clearly-named
agent (Planning Agent / Literature Agent) is running and emitting
output to the panel above. The status-bar wiring on the dashboard
side either isn't subscribed to the right SSE event class or
isn't reading the `agent` / `model` fields off the message it
already receives.

**First diagnostic:** check the SSE stream emitted by
`/api/projects/<n>/runs/<id>/stream` — does it include `agent` /
`model` markers, and does the dashboard's status-bar JS bind to
them? The Textual TUI's status bar binds via the
`ThinkingPanel.set_model` / `panel.update(agent=...)` path which
we know works (E2E run printed `claude-sonnet-4-5` correctly in
the inner status). Likely the dashboard JS just doesn't read the
same fields.

**Files to check:**
- `src/urika/dashboard/templates/_base.html` (status bar markup)
- `src/urika/dashboard/templates/run_log.html` or equivalent
- `src/urika/dashboard/routers/api.py:event_stream` (the SSE
  emitter — does it forward agent / model markers as `event:
  status` payloads?)
- `_format_log_line` and any helper that translates run-log
  lines into SSE events.

### 2. Prompt-bloat / context-budget for small-context endpoints

**Re-scoped 2026-05-02 after live data collection.** The trace
instrumentation (`URIKA_PROMPT_TRACE_FILE`, committed 7ab30ef8)
showed 88.6% prompt-cache hit ratio across a real run, with output
tokens outweighing fresh input tokens by 200:1. Layer 1 (global
trim) is **not shipping** — it would save tens of fresh input
tokens per turn against a 21K-output-token stream. Layer 2
(per-endpoint `context_window` + `max_output_tokens` declarations)
remains in scope; that's the actual fix for the
ContextWindowExceededError that surfaced this work.

See `2026-05-02-prompt-bloat-and-context-budget.md` (rewritten
2026-05-02) for the full reasoning and the trim-the-meta-
orchestrator-advisor-only follow-up that may surface for v0.4.2.

### 3. SIGTERM during post-criteria report should not exit 1 ✓ DONE (fff7c134)

`stop_session` now refuses to downgrade a terminal status; the
SIGINT/SIGTERM cleanup handler exits 0 with a clarifying message
when it detects the already-completed case. `core/session.py`
records `post_terminal_stop_reason` in the checkpoint so dashboards
can surface "completed but reporting interrupted" later.

### 4. Per-tool-call Bash timeout

Long-running methods (DL training over hours) work today, but a
hung subprocess (infinite loop, deadlocked GPU) hangs the whole
experiment forever — there's no per-bash-call timeout. Add a
configurable `[preferences].max_method_seconds` (default 24h);
when a single Bash tool call exceeds it, fail-the-method and
record a `tool_timeout` observation so the advisor sees it.

### 5. Document checkpoint pattern for long-running training ✓ DONE (b131f013)

Shipped as `docs/21-cookbook-long-running-training.md`. Covers
max_turns sizing, budget calibration, stop / resume semantics
(including the v0.4.1 #3 fix above), the prompt-trace
instrumentation for diagnosing where time goes, and a defensive
metric-write template so silent training crashes surface as
failures.
