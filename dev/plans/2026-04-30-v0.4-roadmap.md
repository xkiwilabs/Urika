# Urika v0.4 — Roadmap

**Status:** active (canonical)
**Date:** 2026-04-30
**Branch:** `dev`
**Last release:** v0.3.2 (`3e093673`, tagged 2026-04-30)

This doc is the single source of truth for v0.4 scope. Every other v0.4
plan in `dev/plans/` either pre-dates this and is referenced below, or is
a per-track design doc this roadmap creates and links to.

The user's stated bar for v0.4: **complete all major features so we can
move to user testing and only fix bugs going forward — no new features
mid-test cycle**. That bar drives the cuts below.

---

## Where v0.3.2 left us

v0.3.2 was a hardening release driven by seven parallel-agent audits.
Net effect:

- **Stop button works end-to-end** (SIGTERM handler + `killpg` + reaper
  terminal-status writeback + real SSE state).
- **Dashboard advisor chat no longer auto-fires experiments** (non-TTY
  guard on `_offer_to_run_advisor_suggestions`).
- **Stale `claude-opus-4-7` settings auto-migrate** on first launch with
  backup at `settings.toml.pre-0.3.2.bak`.
- **`urika new` honors global default model** (was reading legacy flat
  key only).
- **System `claude` CLI preferred over the bundled SDK binary** (which
  lags and rejects newer model schemas).
- **Sessions tab now captures dashboard advisor chats**.
- **Five P0 error-surfacing fixes** across SDK adapter, orchestrator
  loop, dashboard runs, vault writes — every failure now preserves
  type/traceback/cause.
- **Non-TTY guards** on every destructive prompt (advisor, run_planning,
  run, present, report, build-tool).
- **Cross-interface invariant tests** (`tests/test_cross_interface_defaults.py`)
  pin the contracts that pre-v0.3.2 drift broke.
- **`VALID_PRIVACY_MODES` fixed** (was `{open, private, university}`,
  now `{open, private, hybrid}`).
- **Transient + config errors pause-and-resume** rather than failing
  experiments mid-loop.

Released to both `origin/main` and `public/main`, tagged `v0.3.2`.

## What's already shipped

Use this list to avoid re-doing work in v0.4:

- **Secrets vault — feature complete in v0.3.2.** Tiered (process env →
  project `.env` → global keyring/file), sidecar metadata, dashboard
  CRUD on global + per-project tabs, override-from-global flow, paste
  flows in privacy/notifications forms, foot-gun guards, three-section
  Secrets tab with locked-provider rows. CLI parity for global writes.
  All ten `feat(secrets|vault)` commits landed by `3e093673`.
- **Textual TUI v2 — ~85% done.** Three-zone layout, stdin bridge,
  background workers, OutputCapture, animated activity bar, tab
  completion, slash command parity. 56 TUI tests pass. Architecture is
  solid — what's left is polish.
- **Memory + persistence — ~70% built.** `OrchestratorSession` save /
  load / list / prune solid; `/resume-session` wires in TUI/REPL;
  advisor history + rolling context summary working; Sessions tab
  rendering. The locked **project memory** design at
  `2026-04-28-project-memory-design.md` is unbuilt — Phase 1 is the
  v0.4 carry.
- **Multi-provider abstraction — ~60% real.** `AgentRunner` ABC +
  `AgentResult` provider-agnostic; `_make_on_message` callback
  duck-typed; `runtime.backend` plumbed through `RuntimeConfig`. Only
  `ClaudeSDKRunner` exists. Three Anthropic-isms remain to abstract:
  endpoint env-var construction, compliance scrub, tool-name
  vocabulary. See `2026-04-10-agent-runtime-abstraction-design.md`.

## v0.4 scope — five tracks

Total: **27 dev-days** with thin GitHub scope, **44 dev-days** with
thick. Recommendation: thin.

### Track 1: Carry-overs from v0.3.2 deferrals (~6 days)

| Item | Effort | Plan |
|---|---|---|
| **SecurityPolicy enforcement** via SDK `can_use_tool` hook | 4.5d | `2026-04-30-securitypolicy-enforcement.md` |
| **Consistency-sweep PR** for 11 small P1s from the v0.3.2 audits | 1d | This doc + audit findings inline below |
| **Stop endpoints for non-run operations** (advisor / finalize / build-tool / present / summarize) | 0.5d | This doc |

The v0.3.2 CHANGELOG explicitly promised SecurityPolicy enforcement and
the bash allow-list hardening for v0.4. Both close in this track.

**Consistency-sweep contents** (from `2026-04-30-securitypolicy-enforcement.md`
audit + the seven 0.3.2 sweeps):
- Audience default unification (4 sites disagree: `novice` / `standard`
  / `expert`).
- `max_turns_per_experiment` default unification (CLI=5, dashboard=10).
- Email password masked in CLI (`config_notifications.py:452-455` echoes
  plaintext).
- `KNOWN_AGENTS` add `project_summarizer`.
- `VALID_SESSION_STATUSES` add `pending` / `starting`.
- Surface swallowed exceptions in `loop_criteria.py`, `loop.py`,
  `parsing.py`, `cli/_base.py` (UrikaError `__cause__` chain).
- Vault silent-read failures in `_read_env_file` / `_read_meta`.
- `default_max_turns` upper bound (currently no cap).
- `ENDPOINT_CHOICES` derive dynamically from `get_named_endpoints()`.
- CLI notifications wizard stop hardcoding `*_env` names (overwrites
  dashboard-set custom names).
- Dashboard email password vs username clobbering.

### Track 2: Memory + persistence completion (~5 days)

| Item | Effort | Plan |
|---|---|---|
| **Project memory Phase 1** (read + auto-capture via `<memory>` markers) | 3-4d | `2026-04-28-project-memory-design.md` (locked, unimplemented) |
| **Cheap wins bundle** | 0.5d | This doc |
| **`success_criteria` ↔ `criteria.json` unification** | 0.5d | This doc |
| **Phase C orchestrator-memory polish** | 0.25d | This doc |

Phase 2-4 of project memory (curator agent, archive viewer, diff view)
defer to v0.5.

**Cheap wins bundle:**
1. Inject `load_context_summary` into planner + finalizer prompts (the
   pattern already exists at `loop.py:551-568`; copy to other roles).
2. Persist `--instructions` to `projectbook/instructions.md`
   (append-mode); planner reads it the same way it reads
   `advisor-history.json`.
3. `urika sessions export <id> [--format md|json]` plus a Download
   button on the Sessions list page (pure render-from-JSON).

**`success_criteria` ↔ `criteria.json` divergence**: dashboard form
writes `[project].success_criteria` in `urika.toml`, but runtime reads
`criteria.json`. Either route the dashboard write through
`append_criteria(...)` (preferred) or remove the textarea.

### Track 5: GitHub integration (the user-named ask)

**Decision pending: thin vs thick.** Scope spec at
`2026-04-30-github-integration.md`.

| Scope | Effort | Trade-off |
|---|---|---|
| **Thin** (`gh` CLI subprocess, no dashboard auth UI) | ~6-8d | User must have `gh` already authenticated |
| **Thick** (pygit2 + device-flow OAuth + dashboard "Connect" button + queue/retry + audit log UI) | ~24d | 4× the effort but fully self-contained |

**Recommendation: thin for v0.4, thick for v0.5.** Lets us ship the
user-named feature in this cycle without blowing the budget; the thick
path becomes a polish iteration once the surface is stable.

### Track 6 (mini): TUI polish (~1-2 days)

Per `2026-04-24-release-polish.md` Phase 4 (never executed) plus the

- Worker `SystemExit` handlers should bail the command, not exit the
  whole TUI app.
- Esc-key binding inside the Textual app for pause (parity with CLI).
- Regression test for the orchestrator-chat → Bash → urika → SDK

## Explicit deferrals (v0.5+)

Per the cross-cutting feature audit's recommendations:

- Plugin / extension system via entry_points (high value once we have
  third-party tools / agents).
- Project templates (`urika new --template behavioral|timeseries|imaging`).
- PDF/LaTeX export, Jupyter notebook export, arXiv fetcher in
  literature agent, Plotly/Bokeh interactive figures.
- Model-card auto-generation for finalized methods.
- Run replay / decision-log export.
- Optuna hyperopt agent.
- Mobile-responsive dashboard.
- GitHub thick scope (if v0.4 went thin).
- OpenAI Agents SDK adapter end-to-end (the second working backend).
- Memory Phases 2-4 (curator agent, archive viewer, diff view).

## Skip entirely (off-positioning)

Multi-user / collab, telemetry, public-sharing button (use GitHub Pages
instead), CodaLab/BinderHub deep integration, AutoML platforms.

## Implementation order (recommended)

1. **Track 1 first** (~6d). Closes the two SecurityPolicy P0s the
   v0.3.2 CHANGELOG promised, plus the consistency-sweep small wins.
   Lowest risk, highest "feels-done" return.
2. **Track 6 TUI polish** (~1-2d). Unblocks confident user testing of
   the default interface.
3. **Track 4 user-facing features** (~6d) in this order: shell
   completion (trivial) → dataset hash → comparison view → budget
   (most surface-area).
4. **Track 3 thin multi-provider** (~3d). Mostly refactor; can run in
   parallel with Track 4.
5. **Track 5 GitHub thin** (~7d). The user-named feature lands once
   the foundation is solid.
6. **Track 2 project memory Phase 1** (~5d). Land last because it
   touches every agent role's prompt and benefits from a stable base.

After Track 1 ships: cut **v0.4.0-rc1** to start surfacing test feedback
in parallel with the rest. Subsequent tracks ship as `0.4.0-rc2`,
`-rc3`, etc., or land directly into `0.4.0` if the bug count stays low.
