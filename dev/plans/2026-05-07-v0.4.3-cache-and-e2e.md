## Track 1 — prompt-reorder for max cache reuse

### Why

v0.4.2 verified that the bundled `claude` CLI auto-applies
`cache_control: { type: "ephemeral" }` to the last cacheable block.
Live trace: 86% input cache-read rate on a typical advisor call.
Caching is working — but Urika's system prompts have an avoidable
cache-buster at the top.

Current shape (`task_agent_system.md`, lines 5-7 — same pattern in
every agent role):

```
**Project directory:** {project_dir}
**Experiment ID:** {experiment_id}      ← varies per experiment
**Experiment directory:** {experiment_dir}  ← varies per experiment

## Your Mission
... ~5 KB of stable boilerplate ...
```

Because `{experiment_id}` changes between experiments, every
experiment's system-prompt prefix is unique even when the rest of
the boilerplate is byte-identical. That defeats cross-experiment
cache reuse: each new experiment's first agent call re-creates the
~5KB prefix at full input price ($3/MTok on Sonnet, $15/MTok on
Opus) instead of reading it at the 10%-priced cache-read tier.

### What

Move experiment-specific interpolations to the END of every
agent-role prompt. Target shape:

```
## Your Mission
... ~5 KB of stable boilerplate first ...

## Critical: Real Data Only
... another stable section ...

## Instructions
... still stable ...

---

## Project Context

**Project directory:** {project_dir}
**Experiment ID:** {experiment_id}
**Experiment directory:** {experiment_dir}
```

Files to update:

- `src/urika/agents/roles/prompts/task_agent_system.md`
- `src/urika/agents/roles/prompts/data_agent_system.md`
- `src/urika/agents/roles/prompts/planning_agent_system.md`
- `src/urika/agents/roles/prompts/evaluator_system.md`
- `src/urika/agents/roles/prompts/advisor_agent_system.md`
- `src/urika/agents/roles/prompts/finalizer_system.md`
- `src/urika/agents/roles/prompts/report_agent_system.md`
- `src/urika/agents/roles/prompts/presentation_agent_system.md`
- `src/urika/agents/roles/prompts/tool_builder_system.md`
- `src/urika/agents/roles/prompts/literature_agent_system.md`
- `src/urika/agents/roles/prompts/project_builder_system.md`
- `src/urika/agents/roles/prompts/project_summarizer_system.md`
- `src/urika/agents/roles/prompts/orchestrator_system.md`

13 prompt files. Each move is mechanical: extract the top
interpolation block, append it as a "## Project Context" section at
the end. Don't change content — only ordering. Keep the order of
the body sections exactly as they are today.

### Risk

- **Agent behaviour change**: agents read the system prompt
  top-down. Reordering may shift which instructions the model
  weighs more heavily. Mitigation: smoke-test before merging
  (Track 2).
- **Minimum cache token threshold**: Opus 4.7 needs 4096 tokens to
  be cacheable. The reordered prefix must still be ≥4096 tokens or
  caching won't kick in for short-prompt cases. Verify by counting
  tokens on the reordered prompts.
- **Tests pinning the prompt structure**: `test_real_data_prompt.py`
  greps for "Real Data Only" — won't move. But any test that
  checks line numbers or `prompt[:100]` would break. Sweep before
  reorder.

### Effort

~1 hour to reorder + token-count check.

### Validation

Compare before/after via `URIKA_PROMPT_TRACE_FILE` on a multi-
experiment smoke run:

- Before: each new experiment's first agent call has
  `cache_create > cache_read`
- After: most cache_create on the FIRST experiment, subsequent
  experiments mostly cache_read

Aim for >50% reduction in cache_create across experiments 2-N.

## Track 2 — E2E coverage gaps

### Why

v0.4.2 surfaced that the existing `dev/scripts/smoke-v04-e2e-*.sh`
harness only covers the CLI shell surface. Every bug we fixed was
in TUI / REPL / dashboard / interactive-flow code that the smoke
never exercises:

- `cmd_advisor` str/dict (REPL/TUI only — shell `urika advisor`
  is a different code path)
- `/pause` unreachable in TUI
- `/advisor` agent swap to OrchestratorChat
- Lockfile legacy-empty-file handling
- OAuth scrub gaps
- Advisor → /run silent fail

The smoke gave false confidence for two prior releases (v0.4.0,
v0.4.1) — the same harness shipped while these bugs were latent.

### What — split into 4 sub-tracks

#### 2a. REPL E2E via pexpect

Spawn `urika --classic`, send slash commands as bytes, assert on
captured stdout. Standard pattern. Cover the critical flows:

- `/list` → see registered projects
- `/project <name>` → loaded
- `/advisor "what should I try?"` → response captured + suggestions
  populate `pending_suggestions` (would have caught I-1)
- `/run` → picks up pending suggestion (would have caught I-7's
  silent str/dict bug)
- `/pause` during /run → flag written, run pauses (would have
  caught the /pause unreachable bug)
- `/quit` → clean exit

Estimated ~150 LOC. New file: `tests/test_smoke/test_smoke_repl.py`,
marked `@pytest.mark.integration`.

#### 2b. TUI E2E via Textual Pilot + stub agents

Already use `App.run_test()` in `tests/test_tui/test_command_dispatch.py`.
Extend the pattern with a stub `_run_single_agent` that returns
canned responses, then drive:

- App launches with project loaded
- Type free-text → orchestrator returns suggestions → assert panel
  hint + `pending_suggestions` populated
- Type `/run` → assert experiment created from suggestion
- Type `/advisor "test"` → assert advisor_agent invoked (NOT
  OrchestratorChat — would have caught Package I-7)
- Type `/pause` mid-run → assert pause flag written
- Type free-text while `agent_running=True` → assert rejection
  panel hint (would have caught NEW-BUG-2)

Estimated ~300 LOC. New file: `tests/test_smoke/test_smoke_tui.py`,
`@pytest.mark.integration`.

#### 2c. Dashboard E2E via Playwright MCP

Use the new `ui-iterate` / `ui-debug` skills. Spin up a real
dashboard, navigate, interact:

- `/projects` page renders, registered projects appear
- Click project → home page renders
- Settings → Privacy tab → all form fields visible (would have
  caught C9 per-project endpoint fields)
- New Experiment modal opens, validates, submits
- SSE log streaming page connects, shows real lines, closes on
  navigation away
- Compare + Criteria links present in sidebar (H2)
- All vendored static assets serve (htmx, alpine, chart.js — H1)

Estimated ~400 LOC of Playwright actions + screenshots for visual
regression. New: `tests/test_smoke/test_smoke_dashboard.py`,
`@pytest.mark.integration`.

#### 2d. Cheap-config smoke + multi-model coverage

Add a variant of `smoke-v04-e2e-open.sh` that:

- Sets per-agent model overrides via `urika config` before running
  (sonnet for reasoning, haiku for execution)
- Drops `--max-turns` to 3 (open) / 2 (hybrid)
- Adds `URIKA_SMOKE_REAL=1` env-var opt-IN to flip back to opus +
  max-turns=5 for pre-release validation
- `release-to-main.sh` sets `URIKA_SMOKE_REAL=1` before invoking
  the smoke, so canonical release flow gets the expensive variant
  automatically

Estimated ~half day including the release-script wiring.

#### 2e. Backwards-compat fixture suite (cheap addition)

Create `tests/fixtures/legacy/` with:

- `v0.3.0-empty-lockfile/` — pre-v0.3 empty `.lock` file (would
  catch the user-reported "stops with lockfile" bug)
- `v0.3.x-corrupt-progress/` — half-written `progress.json`
- `v0.4.0-no-runtime-block/` — old `urika.toml` shape

Run `urika status`, `urika run --max-turns 1`, `urika unlock` etc.
against each. Would catch backwards-compat regressions trivially.

Estimated ~half day.

### Risk

- **Smoke-harness bugs**: tests of tests can have their own bugs.
  Mitigate by keeping each new harness narrow and deterministic
  (stub agents in TUI/REPL E2E, fixture-driven backwards-compat).
- **Playwright dependency weight**: adds chromium to the dev/CI
  install. Acceptable — already a dependency for the user's
  ui-iterate skill.
- **CI cost**: opus-mode smoke runs ~$3/run. Use the cheap config
  by default (Track 2d) and gate the expensive run on the release
  script.

### Effort total

~3 dev-days for all of Track 2 (a + b + c + d + e).

## Sequencing

1. **Day 1**: Track 2d (cheap-config smoke). Touches
   `dev/scripts/` + `release-to-main.sh`. Lowest risk; immediate
   value (faster dev iteration).
2. **Day 1**: Track 2e (backwards-compat fixtures). Independent of
   the others.
3. **Day 2**: Track 2a (REPL pexpect) + Track 2b (TUI Pilot) — they
   share patterns; can be done together.
4. **Day 3**: Track 2c (dashboard Playwright). Most novel, save for
   when other harnesses are stable.
5. **Day 3 evening**: Track 1 (prompt reorder). Validated by all
   the new harnesses.

After Track 2 lands, every release going forward gets:

- Unit suite (~65 s, runs every commit)
- REPL pexpect smoke (~30 s, runs on PR)
- TUI Pilot smoke (~30 s, runs on PR)
- Dashboard Playwright smoke (~60 s, runs on PR)
- Backwards-compat fixtures (~30 s, runs on PR)
- Cheap-model open-mode E2E (~10 min, runs nightly + pre-release)
- Real-model open-mode E2E (~45 min, runs at release tag time only)

That's the coverage matrix that would have caught everything we
fixed in v0.4.2 — at a tractable cost.

## Out of scope for v0.4.3

- 1-hour TTL via Client SDK fallback (only for specific long-run
  training cases; complex; defer until a user actually hits the
  5-minute TTL ceiling)
- Cache-aware budget estimator (use the broken-out
  `cache_creation_in` / `cache_read_in` fields from Package B to
  show users a "you saved $X via caching" line in the Usage tab)
- Multi-provider E2E (Bedrock / Vertex / Foundry) — defer to v0.5+
  when the multi-provider abstraction is real

These can land later as opportunities arise.
